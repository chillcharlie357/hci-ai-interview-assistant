# 问题、相关工作与解决方案

日期：2026-05-01  
状态：MVP 研究与方案说明

## 1. 本项目尝试解决的问题

本项目要解决的不是“再做一个视频面试平台”，而是补上技术面试中一个更小但更关键的缺口：如何让 AI 辅助完成可控、可复盘、证据清楚的面试提问、回答记录和纪要生成。

现有工作通常已经覆盖了部分能力，但仍有明显空白：

- 企业视频面试产品通常重在招聘流程管理、异步视频筛选、ATS 集成和候选人排序，对“面试问题如何生成、追问为什么发生、纪要结论来自哪段回答”的可解释性不够透明。
- 多模态面试评估论文通常重在表情、语音、视线、姿态等信号建模，但容易走向黑盒评分；对于早期产品来说，直接使用非语言信号做能力判断风险较高，也不利于快速落地。
- 开源 AI 面试项目通常更偏完整 Demo，例如语音 Agent、视频房间或候选人评分，但很多项目把问题生成、提问、回答记录和纪要生成耦合在一起，难以单独复用和审计。
- 传统会议纪要工具能总结对话，但不了解“面试结构”：岗位要求、简历信息、题目维度、追问意图、候选人回答证据和面试官复盘需求。

因此，本项目 MVP 先聚焦一个最小闭环：

1. 输入简历摘要、岗位 JD 和面试目标。
2. 生成结构化面试问题。
3. 数字人面试官逐题提问。
4. 记录候选人回答、回答用时、字数和填充词等基础指标。
5. 面试中记录浏览器端摄像头观察信号和必要关键帧。
6. 面试后生成可追溯的智能纪要。

MVP 暂不做屏幕共享识别、视频抽帧、OCR、高精度表情识别和自动录用判断。摄像头分析只输出可复核观察信号，例如亮度、清晰度 proxy、脸部可见、运动量和活动 proxy，不推断情绪、人格、健康状态或录用结果。

## 2. 相关工作

### 2.1 论文与研究方向

#### 多模态面试评估

相关论文和研究系统通常将视频面试评估建模为多模态问题，输入包括面部表情、视线、语音情绪、语言内容和行为信号。例如：

- [Multimodal AI-Based Mock Interview System](https://www.researchgate.net/publication/374545718_Multimodal_AI-Based_Mock_Interview_System_Integrating_Facial_Expression_Analysis_Speech_Emotion_Recognition_and_NLP_for_Holistic_Candidate_Evaluation)：整合面部表情、语音情绪识别和 NLP，用于模拟面试中的综合评估。
- [IVAS: A multimodal AI system for objective video interview assessment](https://arxiv.org/abs/2407.09155)：强调通过面部情绪、视线和音频分析进行视频面试评估。
- [Multi-Modal Method for Candidate Interview Assessment Based on Computer Vision and Large Language Models](https://www.mdpi.com/2079-9292/13/24/4948)：探索使用计算机视觉与大语言模型进行候选人面试评估。

这些研究说明多模态信号确实有价值，但它们也暴露出 MVP 不应过早进入自动评分的原因：

- 非语言信号受摄像头角度、光照、网络质量、文化差异和个人习惯影响很大。
- 评分模型难以向面试官解释“为什么得出这个评价”。
- 自动化评估容易引入公平性、隐私和合规风险。
- 技术链路较重，不适合作为最小产品的第一依赖。

本项目吸收这些研究的问题意识，但 MVP 选择更保守的路径：先做问题、回答、事件、摄像头观察信号和纪要的证据链。浏览器端非语言分析只作为人工复核线索，不作为自动评分模型。

#### 会议纪要与语音 Agent

会议纪要和语音 Agent 方向已经提供了可复用基础能力，包括 ASR、说话人分离、TTS、对话编排和总结。典型技术包括 Whisper / WhisperX、pyannote.audio、LiveKit Agents、Pipecat 等。

但通用会议纪要并不天然理解技术面试结构。它通常缺少：

- 岗位 JD 和简历之间的匹配关系。
- 题目维度与考察目标。
- 追问意图。
- 每道题的回答证据。
- 面试官用于复盘的结构化输出。

本项目的差异点是把“面试结构”作为一等对象，而不是只把面试当作普通会议录音。

### 2.2 竞品

| 产品 | 主要能力 | 对本项目的启发 | 未充分解决的问题 |
|---|---|---|---|
| [HireVue](https://www.hirevue.com/) | 企业级视频面试、招聘流程、评估与自动化 | 证明企业招聘场景对 AI 面试工具有明确需求 | 更偏企业筛选和流程平台，问题生成与纪要证据链不够透明 |
| [myInterview](https://www.myinterview.com/) | 异步视频面试、候选人筛选、团队协作 | 证明异步面试与候选人体验很重要 | 更偏视频筛选，不是面向可审计的技术面试纪要 |
| [Interviewer.AI](https://interviewer.ai/) | AI 视频面试、候选人评估、招聘自动化 | 证明 AI 面试评分和推荐有市场 | 自动评价逻辑对用户而言可能仍偏黑盒 |
| [Vervoe](https://vervoe.com/) | 技能测试、自动评分、招聘评估 | 证明岗位技能评估可以产品化 | 更偏测评题和自动评分，不是面试过程的证据化记录 |
| [Metaview](https://www.metaview.ai/) | AI 面试记录、转写、总结、招聘系统集成 | 与智能面试纪要方向最接近 | 更偏面试笔记与招聘流程增强，数字人提问和问题生成不是核心 |

竞品整体上证明了三个方向：

- 招聘团队愿意为面试效率和复盘质量付费。
- 面试记录、总结和候选人评估是强需求。
- 自动评分和黑盒推荐需要谨慎处理。

本项目不从“自动筛选候选人”切入，而从“帮助面试官更稳定地提问、记录、复盘，并保存可复核观察信号”切入，降低合规和信任门槛。

### 2.3 开源项目

| 项目 | 地址 | 可复用点 | 局限 |
|---|---|---|---|
| FoloUp | https://github.com/FoloUp/FoloUp | 从 JD 生成问题、AI 语音面试、候选人看板 | 更像完整面试产品，模块边界需要拆解 |
| Flo-Interviewer | https://github.com/buildfastwithai/flo-interviewer | JD / 文档生成面试模板，LiveKit 实时音视频，AI 语音面试官 | 更偏 LiveKit 语音面试 Demo，纪要证据链需要补强 |
| CVQuest | https://github.com/odysa/CVQuest | 从简历生成技术问题和行为问题 | 可作为问题生成参考，但缺少完整面试会话和纪要 |
| Open Interview | https://github.com/dsdanielpark/open-interview | 从简历和 JD 生成技术问答 | 适合参考问题生成，不是完整的面试复盘系统 |
| LiveKit Agents | https://github.com/livekit/agents | 实时语音 / 多模态 Agent，可作为 AI 面试官底座 | 需要自行实现面试业务逻辑 |
| Pipecat | https://github.com/pipecat-ai/pipecat | 实时语音和多模态 Agent 编排 | 强在对话管线，不负责面试结构和纪要 |
| LiveTalking | https://github.com/lipku/LiveTalking | 实时交互数字人和流式渲染 | 适合后续数字人 POC，MVP 不依赖 |

开源项目的价值主要在可复用组件，而不是直接照搬完整系统。本项目会优先复用思想和边界：

- 问题生成可以参考 FoloUp、Flo-Interviewer、CVQuest、Open Interview。
- 后续语音 Agent 可以参考 LiveKit Agents 或 Pipecat。
- 后续数字人可以 POC LiveTalking。
- MVP 仍保持自己的核心数据结构和证据链。

## 3. 本项目的解决方案

### 3.1 设计原则

- 先做最小闭环，再接入重能力。
- 先保证可解释和可审计，再考虑自动评分。
- 先记录问题和回答证据，再生成总结。
- 先用规则模板跑通流程，再用 LLM 增强自然度。
- 摄像头观察只记录指标和关键帧，不解释人格、情绪或能力。
- 保留人工复核入口，不输出录用或不录用结论。

### 3.2 MVP 方案

MVP 采用 Python + TypeScript：

- Python：负责问题生成、会话状态、回答指标统计、video event 接收、纪要生成。
- TypeScript：负责前端 UI、数字人提问展示、前端 TTS、回答输入、浏览器端摄像头观察和结果展示。

核心流程：

```text
简历摘要 + 岗位 JD + 面试目标
  ↓
Python 解析岗位、技能、项目和评估维度
  ↓
生成结构化问题、追问和观察点
  ↓
TypeScript UI 展示数字人提问，并使用前端 TTS 朗读
  ↓
用户录入候选人回答和回答用时
  ↓
TypeScript 记录摄像头观察指标和关键帧事件
  ↓
Python 记录回答、字数、填充词、事件和 video events
  ↓
规则或 OpenAI-compatible LLM 生成 Markdown 智能面试纪要
```

### 3.3 为什么这样做

这个方案刻意避开第一版中高风险、高成本的能力：

- 不依赖真实视频会议，降低工程复杂度。
- 不依赖屏幕共享、OCR 和高精度视觉推理，避免第一版被重视频处理拖慢。
- 实时摄像头分析放在浏览器端，后端只接收指标和内存关键帧，降低部署成本。
- 不依赖真实数字人唇形同步，先用可替换的提问接口验证流程价值。
- 不做自动录用判断，降低合规和信任风险。

同时，它保留了后续扩展点：

- 文本输入可以替换为 ASR。
- 前端 TTS 可以替换为语音 Agent。
- 静态头像可以替换为实时数字人。
- 规则问题生成可以替换为 LLM。
- 规则纪要可以替换为 LLM 纪要生成。
- Canvas proxy 可以替换为 MediaPipe landmark 或更高精度模型。
- 后续可以补充屏幕共享、OCR、视频关键帧和多模态分析。

### 3.4 项目差异点

相比论文，本项目不追求第一版多模态自动评分，而强调面试证据链。

相比竞品，本项目不从黑盒候选人筛选切入，而从面试官可控的提问、记录和纪要切入。

相比开源 Demo，本项目把“面试结构”作为核心抽象：

- 每个问题有维度。
- 每个问题有追问。
- 每个问题有观察点。
- 每个回答有指标。
- 每条纪要结论能回到问题和回答。

## 4. 后续研究方向

- 引入 LLM 生成更贴合岗位的动态追问。
- 引入 ASR，把文本输入改为真实语音回答。
- 接入 LiveKit Agents 或 Pipecat，实现实时语音 AI 面试官。
- POC LiveTalking 或类似项目，实现可替换的数字人提问组件。
- 增加候选人授权、隐私提示和数据保留策略。
- 在充分合规和可解释的前提下，再探索视频、OCR 和多模态分析。
